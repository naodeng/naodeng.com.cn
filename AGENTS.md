# AGENTS.md

## 项目概述

这是一个基于 Astro 的个人内容站点，主站为 `https://inaodeng.com`。仓库不只是放页面模板，还包含博客、Wiki、提示词、工作流、封面生成、内容整理、静态检查、自动化测试和部署相关内容。

这个项目的核心目标不是“把页面跑起来”这么简单，而是稳定维护一个长期更新的双语内容站点。因此做改动时，要同时考虑这几件事：

- 页面是否正常显示，桌面和移动端是否都能用。
- 中文和英文路径、内容、导航是否一致，不要只修一边。
- 内容结构是否还适合被搜索、归档、引用和继续维护。
- 现有脚本、检查项和测试是否会因为改动被破坏。

## 技术栈

- 站点框架：Astro
- 内容格式：Markdown / MDX
- 语言：TypeScript、JavaScript
- 包管理：npm
- 运行环境：Node.js 20 及以上
- 部署：Cloudflare Workers / Wrangler
- 测试：
  - 根目录 `npm run build` 会做站点检查并构建。
  - 根目录 `npm test` 会进入 `tests/` 跑单元测试。
  - `tests/` 目录下还维护了 Playwright 端到端测试。

## 目录认知

开始动手前，先判断改动属于哪一层，尽量在正确位置修改，不要为了“先跑通”把东西塞进错误目录。

- `src/pages/`：页面路由与落地页。
- `src/components/`：可复用页面组件。
- `src/layouts/`：页面骨架和通用布局。
- `src/styles/`：样式。
- `src/utils/`：纯工具逻辑，适合补单元测试。
- `src/blog/`：博客内容，按语言区分。
- `src/content/`：Wiki、文档、提示词、工作流等结构化内容。
- `src/assets/`、`public/`：静态资源。
- `scripts/`：内容处理、检查、生成脚本。
- `tests/`：独立测试工程，不要把测试文件随手塞回主源码目录。
- `docs/`：设计、任务、规范、临时资料。
- `dist/`：构建产物，默认视为结果目录，不作为人工长期编辑入口。

## 工作方式

### 先定义完成标准

接到任务后，先在心里把“完成”具体化，再开始改。至少覆盖下面几项：

- 改动是否真正解决了用户提出的问题。
- 是否影响中文、英文、桌面、移动端中的任一条链路。
- 是否需要补测试、补内容、补脚本检查。
- 是否需要同步更新相关文档或说明。

如果这些问题里还有未确认项，不要急着结束任务。

### 改动前先看上下文

动手前优先看这些信息：

- 相关页面、组件、内容文件和已有实现。
- `README.md`、`tests/README.md`、相关脚本和已有规范。
- 当前目录是否已有相似做法，优先沿用，不要平地起新规则。

### 改动时遵守的原则

- 小步修改，尽量只动和任务直接相关的文件。
- 保持现有命名、目录结构、语言路径和内容组织方式一致。
- 能复用现有组件、布局、脚本时，不重复造一套。
- 涉及双语内容时，主动检查 `en` 和 `zh-cn` 是否都应更新。
- 涉及内容页时，注意标题、摘要、链接、图片、目录、相关推荐等是否还成立。
- 涉及脚本时，优先保证可重复执行，不要留下只适合跑一次的临时逻辑。
- 涉及样式或交互时，至少检查常见视口下是否有明显错位、遮挡、闪动、不可点击问题。

## 验证要求

向用户汇报前，默认必须先验证。不要把“我觉得没问题”当成完成。

### 最低验证标准

按改动类型选择最贴近的验证方式，至少做一项，能多做就多做：

- 改了页面、布局、样式、交互：
  - 运行项目并实际打开页面检查。
  - 至少点一遍相关流程。
  - 看桌面和移动宽度下是否正常。
- 改了工具函数、数据处理、脚本：
  - 用真实或有代表性的输入跑一遍。
  - 检查输出是否符合预期。
  - 试一个边界情况，避免只测“正常值”。
- 改了内容结构、链接、SEO、站点生成逻辑：
  - 至少跑相关检查或构建。
  - 确认没有引入明显报错、空页面、失效链接或路径错误。
- 改了测试本身：
  - 把受影响的测试重新跑过，确认不是“只改断言让它变绿”。

### 常用命令

- `npm run build`
- `npm test`
- `cd tests && npm run test:unit`
- `cd tests && npm run test:e2e`
- `npm run seo:check`
- `npm run wiki:style:check`
- `npm run wiki:integrity:check`

不是每次都要全跑，但必须根据改动挑最相关的命令来证明结果成立。

### 验证失败时的处理

- 先定位是不是自己这次改动引起的。
- 能修就继续修，修完重新验证。
- 如果是仓库里本来就存在的问题，要明确区分“我改坏了什么”和“原本就坏着什么”。
- 只有在确实需要用户决定方向、权限或外部信息时，才带着阻塞点回来问。

## 汇报要求

最终汇报只说对用户真正有用的信息：

- 我做了什么。
- 现在结果怎样。
- 我实际验证了什么。
- 如果还有残留风险，风险具体是什么，会影响什么。

汇报语言要求：

- 用简单直白的中文。
- 不用术语堆砌，不写代码腔，不炫技。
- 默认把用户当作不在看代码的人来解释。
- 先说结果，再补充必要背景。

## 重要约定

- 优先修改源码、内容源文件和脚本，不直接把 `dist/` 当主编辑区。
- 未经确认，不随意改动部署配置、域名相关配置、许可证文本和版权说明。
- 未经确认，不批量改写历史内容的语气、署名、日期、链接结构。
- 新增脚本前先确认仓库里是否已有同类脚本。
- 涉及生成内容时，优先保证可复现，说明输入、输出和触发方式。
- 如果发现仓库里有用户未提交的改动，默认避开，不要清空、覆盖或回退。

## 禁止事项

- 不要在没有验证的情况下直接宣布完成。
- 不要为了省事跳过相关语言版本、相关页面或相关目录的检查。
- 不要把临时调试文件、测试产物、一次性脚本随手提交进仓库。
- 不要随意修改 `dist/`、`node_modules/`、测试报告等生成目录来“伪修复”问题。
- 不要未经确认执行高风险操作，比如批量删除、强制回退、覆盖用户现有修改。
- 不要为了让检查通过而删除测试、放宽断言、绕开真实问题。
- 不要编造“已经验证”“已经跑过”“页面正常”这类结论，没做就明确说明没做。

## 默认交付清单

除非任务本身很小且明显不需要，否则交付前至少自查下面这些项：

- 相关文件已更新，且修改点和任务一致。
- 没误伤无关内容。
- 跑过至少一种有效验证。
- 验证结果已看过，不是只执行命令。
- 最终汇报能让不看代码的人也明白结果。

## 用户补充要求

向我汇报结果时，用简单直白的语言说清楚你做了什么、结果怎样。最终回复里不要出现术语、技术实现细节和代码腔。写的时候就当你在跟一个聪明人解释，但对方没在看代码。你实际干活的过程（怎么思考、怎么规划、怎么写代码、怎么调试、怎么解决问题）该多技术就多技术，该多严谨就多严谨。这条要求只针对你跟我沟通的方式。

在向我汇报之前，尽一切可能先验证自己的工作。不要写完代码就当大功告成。用你手头的工具实际跑一遍。能运行就运行，检查输出，确认它确实做到了我要求的事情。如果你在做 Web 应用这类可视化的东西，打开页面看看，点一遍流程，确认渲染和交互都没问题。如果你在写脚本，拿真实的或有代表性的输入跑一遍，检查结果。如果有能模拟的边界情况，试一试。

开始之前，先给自己定义好完成标准：这个任务做到什么程度才算'完成'？把它当作交付前的检查清单。如果有什么跑不通或者看着不对，修好，重新测。不要只是标记一下问题就甩回给我。目标是不让我参与迭代过程。我要收到的是完成的、能用的成果，不是需要我再去逐项检查的初稿。只有在你确认一切正常，或者确实遇到了需要我介入的障碍时，才回来找我。
