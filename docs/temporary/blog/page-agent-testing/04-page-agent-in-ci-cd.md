# Page Agent 在软件测试中的应用：CI/CD

## 文章定位

- 系列：Page Agent 在软件测试中的应用
- 状态：发布版
- 主题：CI/CD

## 导读

这篇讲 Page Agent 如何接入持续交付链路，把“本地能跑”升级为“流水线可依赖”的质量能力。

## 一、为什么要接入 CI/CD

如果 Page Agent 只停留在本地验证，价值会被限制在个人效率层。接入 CI/CD 后，它才能成为团队级质量信号来源。

接入目标应当明确为三点：

- 让关键场景在提交阶段尽早反馈
- 让版本回归有稳定执行与证据沉淀
- 让失败结果可用于阻断和修复优先级决策

## 二、接入方案设计

建议分两类任务接入。

第一类：快速反馈任务（PR 阶段）。

- 只跑少量高价值冒烟场景
- 目标是快，不是全

第二类：稳定回归任务（主干与定时）。

- 覆盖主流程与跨页面关键链路
- 结合可选 Chrome 扩展处理多标签页任务

如果你们流程跨页面较多，扩展能力应在方案设计期就评估，不要等失败后再补。

## 三、流水线中的关键实践

实践 1：双通道产出。

- 给机器：结构化执行结果，用于门禁判断
- 给人：可读执行过程，用于故障复盘

实践 2：失败重试要有边界。

重试可以降噪，但不能掩盖真实不稳定。建议限制重试次数，并记录首次失败原因。

实践 3：把“可测性改造”纳入 CI 目标。

当页面频繁变更导致任务波动时，不能只改脚本，要反向推动页面可测性规范。

## 四、常见问题与治理

问题 1：流水线时长膨胀。

- 治理：分层执行，冒烟前置，全量后置。

问题 2：通过率不稳定。

- 治理：优化语义描述、关键断言和页面状态等待策略。

问题 3：结果能看不能用。

- 治理：建立最小指标体系，如通过率、误报率、定位时长、阻断有效性。

## 五、总结

CI/CD 不是“把场景塞进流水线”，而是“把结果变成决策”。Page Agent 接入后，真正要追求的是质量反馈更早、复盘更快、阻断更准。

做到这三点，它才是交付体系资产，而不是展示型工具。
