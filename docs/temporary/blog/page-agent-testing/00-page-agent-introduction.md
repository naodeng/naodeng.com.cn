# Page Agent 在软件测试中的应用：介绍

## 文章定位

- 系列：Page Agent 在软件测试中的应用
- 状态：发布版
- 主题：介绍

## 导读

这篇是系列开篇，重点讲清三件事：Page Agent 是什么、它和传统自动化方案有什么本质差异、它在测试团队里最适合放在什么位置。

## 一、为什么要关注 Page Agent

Page Agent 的官方定位很直接：The GUI Agent Living in Your Webpage。简单说，它把“自然语言驱动界面操作”放进页面内 JavaScript 里执行。

这个定位对测试团队有两个现实意义。

第一，它不是一套外置的重型自动化链路。官方明确强调无需浏览器插件、Python 或无头浏览器就能在页面内完成主要操作。对前端测试和产品内测场景来说，接入门槛会更低。

第二，它走的是基于文本的 DOM 操作路线，而不是依赖截图和多模态模型。对于受限环境、权限严格或成本敏感的团队，这条路线更容易做工程落地。

## 二、没有 Page Agent 时通常怎么做

在没有 Page Agent 时，测试团队做 GUI 自动化通常有两条主线：

- 工程化脚本线：Playwright / Cypress / Selenium + 选择器与断言
- RPA 或插件线：外部工具驱动界面流程

这两条线都能做，但常见痛点也很典型：

- 场景表达偏“技术描述”，业务同学读起来不直观
- 复杂后台系统步骤长，脚本维护成本高
- 团队想做产品内 AI 助手时，落地路径和测试路径是割裂的

Page Agent 之所以值得关注，不在于“替代一切”，而在于它补上了一个中间层：自然语言可读性 + 页面内可执行性。

## 三、引入 Page Agent 后会发生什么变化

引入后最明显的变化有三类。

第一类是交互方式变化。

从“先写细粒度脚本”转向“先表达目标动作，再做关键节点校验”。这会让测试步骤更接近业务语言，评审效率更高。

第二类是产品化路径变化。

官方把 SaaS AI Copilot、智能表单填写、无障碍增强列为核心用例，这意味着它不仅能用于测试执行，也能用于你自己产品里的交互增强。测试和产品能力可以共用一套技术资产。

第三类是多页面能力变化。

Page Agent 提供可选 Chrome 扩展支持跨页面任务。对于涉及多个标签页的业务流程，这给了团队一个渐进扩展路径。

## 四、适用边界与风险提示

适合优先尝试的场景：

- 管理后台、ERP/CRM 这类表单与流程密集系统
- 希望在产品内落地 AI 操作助手并同步建设测试能力
- 需要自然语言驱动与人机协同调试能力

不建议一上来就重押的场景：

- 对底层浏览器行为或协议级能力强依赖的场景
- 需要服务端大规模并发自动化执行的场景
- 团队没有统一可测性规范、断言标准与结果复盘机制

另外要注意官方边界：Page Agent 设计目标是客户端网页增强，不是服务端自动化工具。这个边界在方案选型阶段就应写进技术决策。

## 五、总结

Page Agent 的价值不是“让自动化更炫”，而是让界面自动化更贴近业务语言、更容易嵌入产品、更容易用小成本启动。

更稳妥的落地方式是：先从 1 条高价值流程试点，跑通后再做稳定性和规模化治理。它适合作为测试工程体系中的增量能力，而不是一次性重构替代方案。
