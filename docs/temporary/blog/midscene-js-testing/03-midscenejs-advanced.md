# Midscene.js 在软件测试中的应用：进阶

## 文章定位

- 系列：Midscene.js 在软件测试中的应用
- 状态：发布版
- 主题：进阶

## 导读

这篇讨论“规模化前夜”的核心问题：如何通过分层设计、稳定性治理和执行策略优化，把自动化从可用推进到可持续。

## 一、进阶目标定义

当团队把 Midscene.js 用到第二个月，通常会遇到一个转折点：脚本数量增长了，但问题也开始集中出现，比如偶发失败增多、调试时间变长、执行成本上升。这时候如果还用入门阶段的方式维护，效率会明显下降。

进阶阶段的目标不再是“能跑”，而是“稳定、可扩展、可治理”。可以用一句话概括：把脚本从个人技巧升级为团队工程。

建议在这一阶段明确三个衡量标准：

- 失败可归因：每次失败能快速判断是环境、数据、脚本还是模型波动。
- 用例可分层：冒烟、主流程、深度回归有清晰边界，不再一锅跑。
- 结果可追踪：既能看单次报告，也能看趋势变化和风险分布。

## 二、架构与工程化优化

进阶阶段最重要的动作，是给测试脚本建立“分层结构”。如果所有步骤都写成同一粒度，脚本要么可读性差，要么稳定性差。

推荐按三层组织动作：

第一层：意图层（高层目标）。

使用 `aiAct` 描述完整业务意图，例如“以普通用户身份完成下单并进入订单详情”。这一层适合表达业务价值和流程目标。

第二层：控制层（关键节点）。

对易波动、强依赖的步骤使用 `aiTap`、`aiInput`、`aiScroll`、`aiWaitFor` 等原子能力，确保关键路径可控。

第三层：验证层（质量闭环）。

使用 `aiAssert`、`aiBoolean`、`aiQuery` 对关键结果做验证，避免“执行了动作但没验证结果”的假成功。

这种分层的好处是：既保留自然语言表达优势，又把风险集中在可控节点，后续维护时可以精准改动，不需要整段重写。

## 三、稳定性治理

进阶阶段最容易被忽视但最有价值的工作，就是稳定性治理。建议把治理动作固定成流程，而不是临时救火。

治理动作 1：规范步骤语义。

把模糊描述替换为带业务语义的描述，减少理解歧义。例如从“点继续”改成“点击‘提交订单’并进入支付确认页”。

治理动作 2：统一等待策略。

禁用随意 sleep，改用 `aiWaitFor` 或状态化等待，让等待行为和业务状态绑定，而不是和时间常量绑定。

治理动作 3：失败分类与处置。

建议建立最小分类集：环境问题、数据问题、步骤问题、断言问题、模型不确定性。每类问题配一条处置策略，长期看会显著降低重复故障。

治理动作 4：报告资产化。

不要只保存“是否通过”，而要保存可回放报告与必要的结构化汇总。单次报告用于定位，汇总数据用于趋势和优先级决策。

## 四、效率提升实践

随着脚本规模增长，效率提升不能再靠“写得更快”，要靠“执行策略更合理”。

实践 1：分层执行。

把用例按风险和价值分层：PR 阶段只跑高价值冒烟，主干合并跑主流程，夜间任务跑全量回归。

实践 2：并发与批量执行。

在 YAML 运行器中合理使用 `--files`、`--concurrent`，让执行资源与任务复杂度匹配，避免串行堵塞。

实践 3：错误容忍与上下文复用。

对于长链路场景，可根据需要使用 `--continue-on-error`、`--share-browser-context`，提升任务完成率并减少重复初始化成本。

实践 4：变量化和模板化。

把高频流程抽象成模板，数据走变量注入。这样需求变化时改模板即可，不必每个脚本都手工调整。

## 五、总结

进阶阶段的本质，是把 Midscene.js 从“自动化能力”变成“质量生产力系统”。

当团队做到这些迹象，就说明进阶有效：

- 失败定位时间持续下降。
- 新脚本平均开发时间稳定可控。
- 回归噪音下降，质量信号更可信。
- 自动化结果开始反向驱动需求和设计改进。

接下来的重点就是把这些能力接入 CI/CD，让自动化真正进入交付主流程。
