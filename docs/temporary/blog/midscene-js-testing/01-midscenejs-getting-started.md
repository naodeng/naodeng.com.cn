# Midscene.js 在软件测试中的应用：入门

## 文章定位

- 系列：Midscene.js 在软件测试中的应用
- 状态：发布版
- 主题：入门

## 导读

这篇聚焦“第一条可运行链路”怎么搭：环境、模型配置、最小示例、常见坑位和入门阶段的稳定化实践。

## 一、准备工作

如果把 Midscene.js 的学习过程比作搭建一条测试流水线，入门阶段的目标只有一个：尽快建立最小可运行闭环。不要一开始就追求覆盖率，也不要先讨论“全量替换”，先让团队看到它怎么真正解决问题。

建议第一步从 Web + Playwright 开始，原因很简单：资料完整、落地快、反馈直接。你需要准备三类基础条件。

第一类是运行环境。

- Node.js 与包管理器可用。
- 本地可运行 Playwright 项目，或先新建最小项目。
- 团队统一基础脚本命令，避免“每个人一套启动方式”。

第二类是模型配置。

```bash
export MIDSCENE_MODEL_BASE_URL="https://your-model-service/v1"
export MIDSCENE_MODEL_API_KEY="your-api-key"
export MIDSCENE_MODEL_NAME="your-model-name"
export MIDSCENE_MODEL_FAMILY="your-model-family"
```

第三类是测试数据和环境约束。

- 先准备稳定账号与可重复数据。
- 明确测试环境刷新策略。
- 约定失败截图、日志、报告存放位置。

很多团队把入门失败归因于“工具不好用”，其实更常见的原因是环境和数据不稳定。

## 二、第一个可运行示例

第一个示例不要复杂，最好只覆盖一条短链路，但必须包含“动作 + 断言 + 报告”三个环节。推荐最小路径如下：

1. 安装依赖：`@midscene/web`、`playwright`、`@playwright/test`、`tsx`。
2. 新建 `demo.ts`，初始化 `PlaywrightAgent`。
3. 执行 2-3 个动作，例如打开页面、输入关键词、触发搜索。
4. 用 `aiAssert` 验证关键结果。
5. 输出并查看报告，确认失败时能回放关键步骤。

这一步的核心不是“写出很炫的脚本”，而是验证四件事：

- 你能稳定执行动作（`aiAct` 或原子动作）。
- 你能准确表达预期（`aiAssert`）。
- 你能提取关键信息（`aiQuery` / `aiString` / `aiNumber`）。
- 你能用报告解释失败原因，而不是靠猜。

如果这四件事打通，入门就算成功一半。

## 三、常见入门坑位

入门阶段最容易踩坑的不是 API，而是“测试上下文失控”。

坑位 1：模型配置错误。

环境变量拼写、模型名称、服务地址、权限范围只要有一个不对，就会直接导致执行异常。建议团队统一 `.env` 模板，并在脚本启动前做配置校验。

坑位 2：页面稳定性判断不足。

页面还没完成加载就开始执行动作，结果就是偶发失败。比起固定 sleep，更建议使用 `aiWaitFor` 或明确状态判断。

坑位 3：步骤描述太泛。

像“点击那个按钮”“进入下一个页面”这类描述，AI 难以稳定理解。建议加入业务语义和目标标识，让动作表达更具体。

坑位 4：只关注执行，不做复盘。

很多团队脚本失败后只重跑，不追因，最后会形成“偶发失败常态化”。正确做法是每次失败都看报告，并标注失败类型：环境、数据、步骤、断言、模型波动。

## 四、入门阶段最佳实践

要让入门走得稳，建议遵循“先控制，再提效”的顺序。

实践 1：关键节点优先用原子动作。

先用 `aiInput`、`aiTap`、`aiWaitFor` 把关键跳转点和风险步骤固定住，再逐步加入 `aiAct` 提高编写效率。

实践 2：每条主链路至少有一个业务断言。

不要把“流程跑完”当成功，必须检查关键业务结果，比如状态变化、提示信息、核心字段是否正确。

实践 3：把报告当资产，而不是附件。

每次执行结果都保存，失败样本做分类沉淀。两周后再看，会发现最常见的问题往往集中在少数模式里。

实践 4：先做小范围模板化。

把第一条成功链路整理成团队模板，包括命名规范、断言规范、失败注释规范。之后新脚本直接复制模板，质量会更稳定。

## 五、总结

入门的里程碑不是“写了多少脚本”，而是“有没有形成稳定闭环”：

- 本地能稳定跑通核心链路。
- 失败能快速定位并复盘。
- 团队对动作分层和断言标准有共同理解。

如果你已经达到这三点，就可以进入下一阶段：从 0 到 1，把单点成功变成可复制的团队方案。
