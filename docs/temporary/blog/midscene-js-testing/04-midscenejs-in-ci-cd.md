# Midscene.js 在软件测试中的应用：CI/CD

## 文章定位

- 系列：Midscene.js 在软件测试中的应用
- 状态：发布版
- 主题：CI/CD

## 导读

这篇把焦点放在交付链路：如何让 Midscene.js 从“本地可跑”升级为“流水线可依赖”，并输出可用于决策的质量信号。

## 一、为什么要接入 CI/CD

很多团队在本地已经能把 Midscene.js 跑得不错，但如果它没有进入 CI/CD，价值会被大幅削弱。原因很简单：质量反馈只有进入交付主链路，才能真正影响发布风险。

把 Midscene.js 接入 CI/CD，本质上是在做三件事：

- 把质量检查前移到 PR 阶段。
- 把回归验证标准化到主干和发版流程。
- 把测试失败证据沉淀成可追踪数据。

从组织收益看，最关键的不是“是否拦截失败”，而是“是否让问题更早、更准、更可解释”。

## 二、接入方案设计

接入时建议分两条路径并行推进。

路径 A：Playwright 项目集成。

如果团队已有 Playwright 基础，可以优先在现有测试项目中引入 Midscene.js。常见配置点包括：

- 在 `playwright.config.ts` 调整超时与重试策略（比如更合理的 90 秒级超时）。
- 接入 `@midscene/web/playwright-reporter` 生成可回放报告。
- 区分 `merged` 与 `separate` 报告模式，按团队复盘习惯选择。

路径 B：YAML/CLI 任务接入。

如果团队需要快速覆盖或低门槛扩展，可以使用 `midscene` 命令执行 YAML 任务：

- 用 glob 批量指定脚本。
- 用 `--summary` 输出 JSON 汇总，交给流水线门禁和看板系统。
- 用环境变量注入测试数据，减少脚本分叉。

这两条路径并不冲突，通常可以“核心流程走 SDK、批量场景走 YAML”。

## 三、流水线中的关键实践

实践 1：分层触发策略。

- PR：只跑高价值冒烟，保证反馈速度。
- 主干：跑主流程与关键回归。
- 夜间：跑大规模全量任务，做趋势观察。

实践 2：双通道结果输出。

- 给机器：JSON 汇总，用于门禁与告警规则。
- 给人：HTML 报告，用于失败复盘与跨角色沟通。

实践 3：失败复现机制。

线上失败必须可复现。建议保留必要参数和上下文，在排障时可切换 `--headed` 或 `--keep-window` 进行可视化复盘。

实践 4：执行资源治理。

并发并非越大越好。应根据任务类型、页面复杂度、环境容量动态调节 `--concurrent`，避免资源争抢导致假失败。

## 四、常见问题与治理

问题 1：流水线变慢，研发抱怨反馈迟。

治理建议：分层拆分任务，PR 只保留最关键链路；把重任务放到合并后或定时执行。

问题 2：误报太多，团队逐渐不信任自动化。

治理建议：高风险节点优先原子动作，断言必须业务化，统一等待策略，持续清理高波动脚本。

问题 3：跨脚本重复登录和初始化，执行成本高。

治理建议：合适场景启用 `--share-browser-context`，并配合账号隔离和状态复用策略。

问题 4：结果很多，但没有质量决策价值。

治理建议：定义最小质量指标（通过率、误报率、定位时长、阻断有效性），让数据进入发布评审，而不是停留在测试报告页。

## 五、总结

CI/CD 不是把脚本放进流水线这么简单，而是把自动化结果变成交付决策的一部分。

当 Midscene.js 在 CI/CD 中稳定运行后，你会看到三个变化：

- 问题发现更早，回滚和线上救火更少。
- 失败定位更快，沟通成本更低。
- 质量门禁更可信，团队更愿意依赖自动化结果。

从这个阶段开始，自动化不再是“测试环节任务”，而是“交付体系能力”。
