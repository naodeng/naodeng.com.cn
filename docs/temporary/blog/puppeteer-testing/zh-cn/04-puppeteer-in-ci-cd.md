# Puppeteer 在软件测试中的应用：CI/CD

## 文章定位

- 系列：Puppeteer 在软件测试中的应用
- 状态：发布版
- 主题：CI/CD

## 导读

这篇聚焦一个很现实的问题：Puppeteer 在本地跑得通，不代表进了流水线也会稳。真正要做好，核心是把浏览器、配置、隔离和失败处理都前移到流水线设计里。

## 一、为什么要接入 CI/CD

把 Puppeteer 接进 CI/CD，价值不只是“自动跑一下”。

- 能更早发现页面回归问题。
- 能把浏览器层面的验证纳入交付门槛。
- 能让本地验证和发布前验证更接近。

如果一套浏览器验证只停留在个人电脑上，它的团队价值会很有限。

## 二、接入方案设计

结合官方资料，接入方案至少要先定三件事：

- 浏览器来源怎么管。
- 配置放在哪里。
- 不同任务之间如何隔离。

官方配置指南建议优先使用配置文件；如果改动涉及浏览器下载选项，还要重新执行浏览器安装步骤。对流水线来说，这等于在提醒我们：浏览器准备本身就是交付链路的一部分，不能只看测试脚本。

## 三、流水线中的关键实践

在 CI/CD 里，Puppeteer 更适合遵循这几条原则：

- PR 阶段只跑少量高价值流程，先保证反馈速度。
- 主干或定时任务再扩大覆盖面。
- 用独立上下文控制会话隔离，避免流程相互污染。
- 统一保留失败时的截图、日志或其他可排查结果。

如果使用容器环境，官方资料也提供了 Docker 方向的建议，这对需要统一浏览器运行环境的团队很有帮助。

## 四、常见问题与治理

流水线里最常见的问题通常不是脚本语法，而是环境偏差：

- 本地能下载浏览器，流水线里却没有正确准备。
- 本地能看到页面，流水线里却没提前处理无界面差异。
- 多条任务共用状态，导致结果互相影响。
- 出错后没有保留足够信息，导致只能重跑猜原因。

这类问题一旦出现，治理重点不是“再补几个等待”，而是回到环境一致性和结果留存上。

## 五、总结

Puppeteer 接入 CI/CD 的关键，不是把脚本搬进去，而是把浏览器运行条件一起工程化。浏览器准备、配置管理、会话隔离和失败留痕，这四件事做好了，流水线里的结果才值得信。
