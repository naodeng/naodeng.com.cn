# Puppeteer 在软件测试中的应用：入门

## 文章定位

- 系列：Puppeteer 在软件测试中的应用
- 状态：发布版
- 主题：入门

## 导读

这篇以官方入门资料为主线，目标很简单：把 Puppeteer 跑起来，知道浏览器怎么来、脚本怎么写、第一步该避开什么坑。

## 一、准备工作

官方首页给出的第一步非常直接：安装 `puppeteer` 时会下载兼容版本的 Chrome；如果你只想把它当库来连现有浏览器，则可以装 `puppeteer-core`。这意味着入门前要先想清楚一件事：你是希望“开箱即用”，还是“接已有浏览器环境”。

入门前建议准备好三样东西：

- Node.js 环境。
- 一个稳定可访问的测试页面。
- 明确选择 `puppeteer` 还是 `puppeteer-core`。

## 二、第一个可运行示例

最小可运行路径其实不复杂：

1. 启动浏览器。
2. 新建页面。
3. 打开目标地址。
4. 做一个简单动作或检查。
5. 关闭浏览器。

官方浏览器管理指南就是按这个节奏展开的。对测试新人来说，最关键的不是一开始就写很多断言，而是先确认“浏览器能起、页面能开、动作能执行、结果能拿到”。

## 三、常见入门坑位

新手阶段最容易踩的坑，基本都和环境和预期不一致有关：

- 以为安装完就万无一失，却没留意浏览器下载或配置是否真的生效。
- 搞混 `puppeteer` 和 `puppeteer-core`，导致期望它自动下载浏览器，但实际不会。
- 脚本跑不通时只盯页面，不去看调试输出。
- 本地用有界面模式，流水线里却跑无界面模式，结果前后表现不一致。

## 四、入门阶段最佳实践

结合官方资料，入门期更稳的做法是：

- 优先从最短流程开始，不要一上来就做超长业务链路。
- 调试时先用可见浏览器和慢速执行，确认每一步到底发生了什么。
- 尽早把浏览器启动、页面打开、收尾关闭写成固定结构。
- 如果改了下载相关配置，记得按官方建议重新执行浏览器安装步骤，不要以为改了配置文件就会自动生效。

## 五、总结

Puppeteer 的入门门槛不高，但真正决定你能不能顺利上手的，不是 API 数量，而是能不能先把“浏览器、脚本、调试”这条最小链路跑通。先跑通，再扩展，效率会高很多。
