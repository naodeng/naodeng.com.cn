# Cypress 在软件测试中的应用：CI/CD

## 文章定位

- 系列：Cypress 在软件测试中的应用
- 状态：发布版
- 主题：CI/CD

## 导读

这篇聚焦三个问题：为什么要把 Cypress 接进流水线、接入时最容易出什么问题、以及怎样让执行结果真正服务发版而不是制造噪音。

## 一、为什么要接入 CI/CD

只在本地跑 Cypress，价值是有限的。它能帮个人调试和验证，但还不能真正变成团队的质量关口。

一旦接进流水线，意义就完全不同了：

- 代码每次提交后都能自动验证核心流程。
- 团队能更早发现明显回归，而不是等到提测后再集中暴露。
- 失败结果可以留痕，方便复盘和对比。

官方文档对 CI 的定位也很直接：在持续集成里运行 Cypress，基本上和本地终端执行很像，核心就是安装和运行，只是要把环境、服务启动和结果记录处理好。

## 二、接入方案设计

官方给出的最基本接入方式非常简单：

```bash
npm install cypress --save-dev
npx cypress run
```

但真正让流水线稳定的，不是这两行命令，而是前后配套。

最关键的一步，是先把被测服务正确启动起来。官方专门提醒不要直接写成 `npm start & cypress run` 这种方式，因为服务是否已经准备好并没有保证。页面没起来，测试就先跑了，失败其实不是测试问题，而是启动时机问题。

官方给出的更稳做法，是使用 `wait-on` 或 `start-server-and-test` 这样的方式，等到服务真的可访问之后再执行测试。这一步非常重要，因为它解决的是流水线里最常见的“偶发失败”根源之一。

如果团队计划进一步扩大规模，官方还提供了几条可选路径：

- 录制到 Cypress Cloud。
- 多机器并行执行。
- 使用官方 Docker 镜像来固定运行环境。

这些能力不是第一天就必须上，但它们给了团队后续扩容空间。

## 三、流水线中的关键实践

把 Cypress 接进 CI 后，真正该盯住的是下面几件事。

第一，环境一致性。官方建议使用 Cypress Docker 镜像来获得更稳定的一致环境，避免 CI 平台升级 Node 或浏览器版本后，测试表现突然变化。

第二，服务可用性检查。不要靠固定睡眠时间赌服务已经启动，应该用明确的“服务可访问”作为测试开始条件。

第三，结果可追踪。如果团队接入 Cypress Cloud，官方说明可以录制运行结果、看失败截图和视频，还能在并行执行时看到不同机器分担了哪些测试。这些能力对定位问题和评估执行效率都很有帮助。

第四，执行分层。不是每次都要把所有测试全跑一遍。真正有效的做法通常是先跑高价值、稳定度高的核心集，再根据分支、阶段或风险程度决定是否扩展更多测试。

## 四、常见问题与治理

CI 里最常见的问题，往往不是 Cypress 自己“跑不动”，而是接入方式不稳。

常见风险包括：

- 服务没启动好，测试已经开始。
- CI 机器资源不足，导致运行中断、视频异常或执行时间明显拉长。
- 本地能跑，CI 不能跑，因为浏览器或依赖版本不一致。
- 想提速却没有做好分层，结果把大量低价值场景也塞进阻塞发版的流程里。

这些问题里，前两个官方都已经给了很明确的提醒。比如安装文档就提到 CI 至少要有基本的 CPU 和内存条件，持续集成文档也反复强调服务启动和环境一致性问题。

## 五、总结

把 Cypress 接进 CI/CD，不只是“把命令搬到流水线里”。真正有效的是把服务启动、环境一致性、结果记录和执行分层一起做好。

如果团队已经做到这一步，Cypress 的价值就不再只是本地调试工具，而是能直接影响提交质量和发版节奏的质量基础设施。最后一篇，我们再把视角拉宽，看看除了常规 UI 回归之外，Cypress 还能扩展到哪些更有意思的使用场景。
