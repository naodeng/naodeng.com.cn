# Cypress 在软件测试中的应用：入门

## 文章定位

- 系列：Cypress 在软件测试中的应用
- 状态：发布版
- 主题：入门

## 导读

这篇聚焦三个实际问题：怎么把 Cypress 安装起来、怎么跑通第一个测试、以及入门阶段最容易踩哪些坑。

## 一、准备工作

按照官方安装文档，入门第一步其实很直接：确认环境、装包、打开应用。

官方当前给出的基础要求包括：

- 使用受支持的操作系统。
- Node.js 使用 20.x、22.x 或 24.x 及以上版本。
- 通过 npm、Yarn 或 pnpm 安装。

安装命令也很简单：

```bash
npm install cypress --save-dev
```

安装完成后，用下面的命令打开 Cypress App：

```bash
npx cypress open
```

官方把这一步设计成非常清楚的分流入口。你打开之后，可以选择做端到端测试，或者做组件测试。这个设计对入门者很友好，因为它一开始就把“你要测什么”摆在台面上，而不是让你在一堆配置里迷路。

## 二、第一个可运行示例

如果你只是想尽快建立感受，最好的目标不是“写一个复杂业务流程”，而是先跑通一个最小示例。

官方的入门路径很明确：

1. 在项目里安装 Cypress。
2. 打开 Cypress App。
3. 选择 E2E 或组件测试。
4. 写并运行第一个测试。

对于很多 Web 项目来说，第一条测试可以非常朴素，比如访问页面、确认关键信息存在、执行一个最基本交互。这里的重点不是覆盖率，而是先确认三件事：

- 项目能正常被 Cypress 拉起或访问。
- 团队机器上能稳定运行。
- 失败时大家知道去哪里看结果、怎么定位原因。

如果是组件测试，官方文档说明 Cypress 对 React、Angular、Vue、Svelte 提供了官方挂载支持。也就是说，前端团队完全可以不等到整站跑起来，再做测试，而是先从单个组件行为验证开始。

## 三、常见入门坑位

入门阶段最容易踩的坑，不是“不会写命令”，而是环境和思路问题。

第一个常见坑，是把它当成纯命令行脚本来理解。其实 Cypress App 本身就是重要的一部分。官方把本地写测试、跑测试、看执行过程都放在这个应用里。如果团队只盯着 `cypress run`，很容易错过它最有价值的调试体验。

第二个常见坑，是环境不匹配。比如 Node 版本不在支持范围、包管理器配置不合适、CI 机器资源不够，都会让入门体验变差。官方安装页还特别提醒了 pnpm、Yarn 的一些注意事项，这说明“装上”不等于“稳定能用”。

第三个常见坑，是一上来就想写很长的完整业务链路。这样一旦失败，你根本分不清是环境问题、选择器问题、页面问题，还是测试本身写得不稳。

第四个常见坑，是把它和原生移动测试混在一起。官方 FAQ 已经说明它不能直接用于原生移动应用测试，所以在目标定义上别一开始就偏了。

## 四、入门阶段最佳实践

如果目标是“快速跑通且不埋雷”，我更建议按下面的顺序起步。

- 先只做一条最短正向路径，确认环境是通的。
- 优先在本地用 App 看执行过程，不要只看命令行结果。
- 如果是前端项目，先决定主战场是 E2E 还是组件测试，不要两条线一起起。
- 从一开始就约定好元素选择方式，不要靠一堆脆弱定位凑合。

官方最佳实践文档明确建议使用 `data-*` 属性来做更稳定的元素选择，而不是过度依赖容易变化的类名、标签名或文本内容。这一点越早建立习惯，后面返工越少。

另外，官方也不鼓励使用随手写死的等待时间。Cypress 的自动等待和重试能力本来就是为了减少这类写法。如果团队入门时就养成“页面慢了就加 5 秒”的习惯，后面会非常难治理。

## 五、总结

Cypress 的入门门槛不高，但“起步方式对不对”会直接影响后续体验。先把环境对齐、跑通最小示例、用好本地调试、从第一天就约束选择器和等待方式，比急着追求覆盖更多页面重要得多。

当你已经能稳定跑起第一条测试，下一步就该进入真正的落地问题了：怎么从个人试验，走到团队可复制的最小方案。
