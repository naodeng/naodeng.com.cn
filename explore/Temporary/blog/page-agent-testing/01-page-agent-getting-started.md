# Page Agent 在软件测试中的应用：入门

## 文章定位

- 系列：Page Agent 在软件测试中的应用
- 状态：发布版
- 主题：入门

## 导读

这篇聚焦“第一条可执行链路”怎么落地：先快速接入，再跑一个真实测试场景，最后把失败定位和改进节奏建立起来。

## 一、准备工作

Page Agent 的入门门槛不高，但别跳过基础准备。建议先做三件事。

第一，明确接入方式。

官方给了两条主路径：

- 一行 script 方式（用于快速技术评估）
- npm 方式（用于工程化集成）

第二，确定模型接入策略。

官方支持 Bring your own LLM。也就是说，你可以接自己的模型服务，不必绑定单一供应方。入门阶段建议先用一套稳定模型配置跑通主链路，再谈模型切换策略。

第三，准备一个小而完整的测试场景。

不要从最复杂流程开始。优先选择“登录 -> 核心操作 -> 结果校验”这类短链路，确保你能快速闭环。

## 二、第一个可运行示例

官方 npm 示例核心结构很清楚：

- `npm install page-agent`
- `import { PageAgent } from 'page-agent'`
- 初始化 `PageAgent({ model, baseURL, apiKey, language })`
- `await agent.execute('...')`

对测试团队来说，这个最小示例至少要补两件事：

- 关键断言：不只执行动作，还要检查业务结果
- 执行记录：保存步骤与结果，便于失败复盘

一个实践建议：先把自然语言步骤写得“业务可读”，再把关键风控节点转成更明确的验证项。这样既保持可读性，也不会丢稳定性。

## 三、常见入门坑位

坑位 1：把 Demo 能跑当作工程可用。

官方明确说明 demo LLM CDN主要用于技术评估。入门验证通过后，应尽快切到你自己的模型和环境策略。

坑位 2：步骤描述过于模糊。

自然语言不是越短越好。测试场景里建议写清角色、页面目标和成功标准，否则动作不稳定时很难定位。

坑位 3：只关注“执行成功”，不关注“验证有效”。

自动化最常见假象是流程走完了，但业务结果没被验证。入门阶段就要强制加入关键断言。

## 四、入门阶段最佳实践

实践 1：从产品真实高频流程选场景。

你要验证的是“对业务有没有价值”，不是“demo 是否酷炫”。

实践 2：明确人机协同边界。

Page Agent 支持人机协同 UI。建议定义清楚：哪些步骤允许人工介入，哪些步骤必须全自动通过。

实践 3：建立失败分类。

至少区分这四类失败：页面结构变更、数据问题、模型理解偏差、环境问题。两周后复盘会很有价值。

## 五、总结

入门阶段的里程碑很简单：

- 有 1 条可重复执行的核心链路
- 有 1 套最小断言与复盘机制
- 团队知道 Page Agent 在你们体系里该放在哪个层级

达到这三点，再进入“从 0 到 1”的团队化落地，节奏会稳很多。
