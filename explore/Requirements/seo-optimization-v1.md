# SEO Optimization v1（审计增强版）

## 1. 背景

基于现有代码审计，网站已经具备较完整 SEO 基础能力，但仍存在可见性与可维护性风险：

- 已具备统一 Head 能力：`src/layouts/Base.astro` 输出 `title/description/canonical/og/twitter/hreflang/json-ld`。
- 已具备 robots 与 sitemap 基础：`src/pages/robots.txt.ts`、`@astrojs/sitemap`。
- 已具备基础 SEO E2E：`tests/e2e/specs/seo.spec.ts`。
- 仍存在风险：
  - SEO 测试覆盖偏浅，主要是“存在性校验”，缺少“正确性校验”（canonical 唯一性、indexability、结构化数据字段完整性）。
  - `/[lang]/sitemap` 页面展示全量 Wiki 词条，体量大且可能稀释抓取预算。
  - 内容生产规模扩大（Wiki/AIWiki）后，缺少自动化质量门禁（死链、重复标题、描述长度、孤儿页）。

## 2. 目标与结果

### 业务目标
- 提升 Wiki/AIWiki/Blog 的有效收录与稳定排名。
- 提升高曝光页面 CTR（标题与描述更可读、更有区分度）。
- 降低技术 SEO 回归事故（错误 canonical、死链、结构化数据失真）。

### 质量目标（QA 视角）
- SEO 校验从“人工抽查”升级为“CI 门禁 + 周期巡检”。
- 发布后可追踪：出现异常时可回溯到页面、模板、内容来源。

## 3. 范围

### In Scope
- 技术 SEO：canonical/hreflang/sitemap/robots/indexability/结构化数据。
- 内容 SEO：标题与描述规范、内链网络、词条与博客互链。
- QA 工程：自动检查脚本、CI 规则、回归测试与发布检查单。

### Out of Scope
- 付费投放、外链建设、品牌 PR 活动。
- 大规模改写历史文章正文（仅做结构化修复）。

## 4. 审计结论与需求

### R1. 元数据正确性从“存在”升级到“正确”
- 需求：
  - 所有可索引页必须具备唯一且语义准确的 `title/description/canonical`。
  - description 长度统一控制在可读区间（建议 70-155 字符）。
- 验收标准：
  - 缺失率 `= 0`。
  - 重复标题占比 `< 1%`。
  - canonical 与最终 URL 一致率 `= 100%`。

### R2. 抓取与索引治理
- 需求：
  - 建立 indexability 台账（可索引/不可索引及原因）。
  - sitemap 仅包含应被抓取与索引页面。
  - 评估 `/[lang]/sitemap` 展示策略，降低超大页面对 crawl budget 的副作用。
- 验收标准：
  - 非预期 noindex 页面 `= 0`。
  - sitemap 中 4xx/5xx 页面 `= 0`。
  - robots 与 sitemap 规则无冲突。

### R3. 结构化数据质量提升
- 需求：
  - 统一约束 `WebSite/Person/Breadcrumb/Article/DefinedTerm` 字段完整性。
  - 关键字段必须可解析且与页面内容一致。
- 验收标准：
  - 关键模板页面（首页/博客详情/wiki详情/aiwiki详情）结构化数据错误率 `= 0`。

### R4. 内链与信息架构优化
- 需求：
  - Wiki 与 AIWiki 相关词条必须有稳定来源（frontmatter `related` + 双向策略）。
  - Blog 术语页互链建立标准，避免孤儿页。
- 验收标准：
  - 高价值词条孤儿页占比持续下降。
  - 站内关键路径（首页 -> 列表 -> 详情 -> 相关页）完整可达。

### R5. SEO QA 门禁化
- 需求：
  - 将 SEO 检查接入 CI：静态检查 + 页面抽检 + 回归测试。
  - 建立发布前清单与发布后 7/14/30 天复盘机制。
- 验收标准：
  - CI 中 SEO 门禁通过率 `= 100%`（主分支）。
  - 每次发布有 SEO QA 记录与问题闭环。

## 5. KPI

- 收录与覆盖：有效索引页面数量、未索引原因分布。
- 排名与流量：自然搜索点击、展示、CTR、平均排名。
- 技术质量：死链数、重定向链数量、重复标题数、结构化数据错误数。
- 交付质量：SEO 回归缺陷数量（按月）。

## 6. 风险与应对

- 风险：批量改动 metadata 导致排名波动。
  - 应对：分批发布 + 对照组追踪 + 可回滚配置。
- 风险：内容规模增长导致 QA 人工检查不可持续。
  - 应对：脚本化巡检 + CI 强约束 + 每周自动报告。

## 7. 里程碑（建议）

- M1（1 周）：完成基线审计与问题台账。
- M2（1-2 周）：模板与索引治理改造完成。
- M3（1 周）：CI 门禁上线 + 抽检用例补齐。
- M4（持续）：周报与复盘机制稳定运行。

### R3. 结构化数据
- Blog 详情页输出 `Article`，Wiki/AIWiki/Guild 详情页输出 `BreadcrumbList + WebPage`。
- 验收标准：
  - 覆盖率 >= 95%。
  - 结构化数据解析错误率趋近 0（以校验工具和日志为准）。

### R4. 内链网络优化
- 构建词条与教程内容的双向关联，减少孤儿页。
- 验收标准：
  - 重点页面（Top 200）至少 3 条有效站内入链。
  - 孤儿页数量显著下降。

### R5. 观测与 QA 机制
- 建立 Search Console + 技术巡检 + 发布回归闭环。
- 验收标准：
  - 每周固定输出 SEO 健康报告。
  - 发布后 7/14/30 天都有追踪结论和动作项。

## 4. 成功指标（KPI）

- 有效收录页面数提升 >= 30%。
- 目标关键词 Top10 数量提升 >= 25%。
- 平均 CTR 提升 >= 10%。
- 404/重定向链错误下降 >= 80%。
- 关键模板 SEO 检查项（metadata/structured data/canonical）发布前通过率 100%。

## 5. 风险与应对

- 风险：批量改 canonical 导致误收敛。
  - 应对：灰度改造 + 抽样验证 + 回滚开关。
- 风险：结构化数据字段错误影响 rich results。
  - 应对：模板单测 + 发布前校验脚本。
- 风险：过度关键词优化影响可读性。
  - 应对：内容 QA 审核优先“用户可读性 > 关键词密度”。

## 6. QA 验收清单（发布门禁）

- [ ] 可索引页面 metadata 完整。
- [ ] canonical/route/语言版本映射正确。
- [ ] sitemap 与 robots 策略一致。
- [ ] 结构化数据无致命错误。
- [ ] 死链与重定向链在阈值内。
- [ ] 关键页面手工抽检通过（首页、Blog、Wiki、AIWiki、Guild）。
